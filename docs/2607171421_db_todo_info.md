### 5. Зберігання бінарних даних у БД (`bytea`)

`info_block_components_binary` зберігає картинки та файли прямо у БД як `bytea`.

**Проблема:** Це значно збільшує розмір БД, ускладнює бекапи, знижує продуктивність.

**Рекомендація:** Розглянути **гібридний підхід**:
```sql
kbase.info_block_components_binary
    id bigint,
    info_block_header_id bigint,
    info_block_type_component_id bigint,
    file_name character varying(255),   -- оригінальна назва файлу
    mime_type character varying(100),   -- MIME тип
    file_size bigint,                   -- розмір у байтах
    storage_type smallint DEFAULT 1,    -- 1=DB(bytea), 2=FS(path), 3=S3(key)
    value bytea,                        -- для малих файлів / inline
    file_path character varying(500)    -- для зовнішнього сховища (FS/S3)
```
Це дає гнучкість — спочатку зберігати в БД, потім мігрувати до файлової системи або S3 без зміни схеми.

### 7. `show_level` — вузьке рішення для прав доступу

Поле `show_level int` (0=приватний, 1=зареєстровані, 2=публічний) є простим, але не масштабованим. У проекті вже є таблиці `roles` і `privileges`.

**Рекомендація:** Розглянути зв'язок між `sections` та `roles`:
```sql
kbase.section_access  -- таблиця прав доступу до розділів
    section_id bigint,
    role_id bigint,
    access_level smallint  -- 1=read, 2=write, 3=delete
```
Або залишити `show_level` як "базовий рівень" і додати більш гранульовані права пізніше — це також прийнятне рішення для MVP.

### 8. `date_modified_info` у `sections` — можлива денормалізація

Поле `date_modified_info timestamp` в `sections` зберігає час останньої зміни інфоблоків. Це денормалізація для оптимізації (щоб не виконувати JOIN або підзапит до `info_block_headers`).

**Оцінка:** ✅ Прийнятне рішення для кешування/оптимізації, але важливо **підтримувати його актуальність через тригери** або на рівні додатку:
```sql
-- Тригер на UPDATE info_block_headers:
UPDATE sections SET date_modified_info = now() WHERE id = NEW.section_id;
```
